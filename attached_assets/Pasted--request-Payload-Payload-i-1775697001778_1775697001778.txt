但是思考摘要也没返回，不知道是不是我够早的request或者Payload不对

很可能是 **Payload 写法不对**，或者你读错了返回字段；从官方文档看，思考摘要只有在请求里显式开启 `includeThoughts`，并且你去遍历 `response.candidates[0].content.parts`、检查 `part.thought` 时才拿得到。[1]
另外，官方也说明用错 API 版本或把新特性发到旧 endpoint，会直接导致“请求格式看起来能跑，但某些字段不生效”的情况，所以“你发得比较早的 request”这个怀疑是有道理的。[2]

## 最先检查的点

你先看这 4 个地方，通常问题都在这里：

- 是否用了支持该特性的 API 版本；官方明确说，Beta 功能只会在 `/v1beta` 可用，若你用了旧版本或不匹配 endpoint，新字段可能不工作。[2]
- 请求里是否真的把 `includeThoughts` 放进了正确位置；REST 示例里它在 `generationConfig.thinkingConfig.includeThoughts`，SDK 则是在 `config.thinking_config.include_thoughts` 或 `config.thinkingConfig.includeThoughts`。[1]
- 你是不是只读了 `response.text`；官方示例明确要求遍历 `candidates[0].content.parts`，因为 thought summary 和最终答案是分开放在 parts 里的。[1]
- 你是否在流式场景提前结束了读取；官方示例强调要持续读取每个 chunk，并从 chunk 的 `parts` 里找 `part.thought`。[1]

## 你这个情况最像哪一种

如果请求**成功返回答案**，但完全没有 thought summary，最常见就是下面两类：

- `includeThoughts` 没放对层级，服务端忽略了它，于是只给正常答案。[2][1]
- SDK/中间层把 thought parts 吃掉了，你看到的只是封装后的文本；社区里有人报告过 `includeThoughts: true` 开了，但结果被库混成普通 text 或根本没暴露出来。[3][4]

还有一个边缘情况：如果你把 thinking 关掉了，再开 `includeThoughts`，有些模型或工具链会直接报 “include_thoughts is only enabled when thinking is enabled” 之类的错误。[5]

## 正确的 REST 结构

如果你是直接打 REST，结构应类似这样，`includeThoughts` 要放在 `generationConfig.thinkingConfig` 下面。[1]

```json
{
  "contents": [
    {
      "parts": [
        { "text": "解释一下 TCP 三次握手" }
      ]
    }
  ],
  "generationConfig": {
    "thinkingConfig": {
      "includeThoughts": true
    }
  }
}
```

如果你用的是 Gemini 3 Pro，还可以顺手带上 `thinkingLevel`，因为 Gemini 3 Pro 用的是 `thinkingLevel` 控制推理强度，而且官方写明 Gemini 3 Pro 不能关闭 thinking。[1]

## 正确的 SDK 读取方式

不要只看：

```python
print(response.text)
```

而要看：

```python
for part in response.candidates[0].content.parts:
    if not part.text:
        continue
    if part.thought:
        print("THOUGHT:", part.text)
    else:
        print("ANSWER:", part.text)
```

这是官方给出的标准读取方式，因为 thought summary 不保证会出现在 `response.text` 这种便捷字段里。[1]

## 一个排错顺序

按这个顺序查最省时间：

1. 先确认 endpoint 是 `/v1beta/...:generateContent`，不是旧路径。[2][1]
2. 确认 payload 里是 `generationConfig.thinkingConfig.includeThoughts = true`，大小写别错。[1]
3. 打印完整原始 JSON 响应，别只看 SDK 封装字段。[1]
4. 检查 `usageMetadata.thoughtsTokenCount` 是否大于 0；大于 0 说明模型确实思考了。[1]
5. 若 `thoughtsTokenCount > 0` 但 `parts` 里没有 `thought: true`，就要怀疑 SDK 版本、代理层、OpenAI 兼容层或第三方网关吞字段。[3][4]
6. 如果你是多轮对话或函数调用，注意不要错误改写带 `thoughtSignature` 的 parts；Gemini 3 对签名校验更严格，改坏上下文会出问题。[6][1]
